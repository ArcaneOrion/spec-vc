# ADR-001: 严格模式 + [ADR-none] 豁免策略

- **Date**: 2026-04-20
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: process, hook, strictness

## Context

spec-vc 作为"ADR + Commit 双向锚定"的工具化实现,面临一个关键设计张力——**commit-msg hook 的严格度**:

- **完全放任**(不装 hook):ADR 与 commit 的关联依赖开发者自觉,长期必然漂移。对单人项目尤其危险,因为状态差、启动阻力大、注意力分散时,开发者会在 commit message 里省略 `[ADR-NNN]`,几个月后决策链断裂。
- **严格模式**(阻塞无引用的 commit):强制开发者在 commit 之前把决策对应关系想清楚。代价是日常小改动(拼写、格式、依赖 patch 升级)也需要一个引用,容易形成摩擦。
- **分级模式**(只在关键路径严格):需要预先定义"关键路径",配置负担大,且边界容易漂移。

本项目的使用者是 arcaneorion 本人,背景:
- 主项目是 Alice(Agent 项目),迭代频繁,执行状态有明显波动
- 有数字戒断与行动工程的前置问题——低意义、强规训、长延迟的流程容易被放弃
- 但同时,ADR 机制的价值正在于"在决策当下留下记录",松散记录无法实现这个价值

决策时机约束:这是 v0.1 阶段,尚未在任何真实项目中跑通过;**先收紧再松开** 比 **先松散再收紧** 容易——松散改严格需要补历史,严格改松散只是调参数。

## Decision

我们将采用**严格模式**:

1. `commit-msg` hook 强制校验 subject 行必须包含 `[ADR-NNN]` 或 `[ADR-none]`,缺失则阻塞 commit。
2. 引入 `[ADR-none]` 作为**显式豁免标记**,用于不影响架构的改动(拼写、格式、依赖 patch 升级等)。豁免不能通过"省略引用"实现,必须显式声明。
3. 豁免的具体判定由 `src/spec_vc/adr.py` 中的 `exemption_allows` 函数和 `.spec-vc.toml` 中的 `[exemption]` 配置段共同决定,支持路径 pattern、文件扩展名、改动行数阈值等多种规则。项目使用者可按自身项目特点在配置文件中定制严格度,而不是在代码中硬编码。
4. 引用已废弃(`Deprecated`)或被替代(`Superseded by ADR-XXX`)的 ADR 也视为阻塞条件,强制使用者创建新 ADR 或使用 `[ADR-none]` 并在 body 中说明历史上下文。

## Consequences

**积极**:
- commit 之前强制思考"这个改动对应哪个决策",把 spec-vc 从"善意工具"变成"执行协议"
- 阻塞机制与基底神经节的 if-then 协议同构:外部约束帮助启动阶段形成习惯回路,不依赖前额叶在线
- 漂移检测(`/spec-vc adr-status`)在严格模式下更可靠——孤儿 ADR 和幽灵引用的绝对数量受控

**消极**:
- 状态差、启动阻力大时,严格模式会直接阻塞 commit,可能成为临门一脚的摩擦
- `[ADR-none]` 的豁免规则是摩擦调节器——规则太松等于严格模式失效(全用 none 糊过去),规则太紧则日常小改动被阻塞,容易把 hook 禁用掉
- 对于不熟悉本项目的合作者,首次遇到阻塞会困惑(错误信息需要自解释)

**中性**:
- 引入了一层"决策 vs 非决策"的判定负担,但这层判定本来就应当发生——hook 只是把它从隐式变成显式
- `check_exemption` 的实现权被下放给使用者,意味着本 skill 无法对"严格度"做统一保证,只能保证"存在严格度这条边界"

## Alternatives Considered

- **柔性模式**(只在 `prepare-commit-msg` 里插入提示槽位,不阻塞):被排除,理由是"事后提示"无法对抗启动阶段的注意力分散,与 arcaneorion 的实际工作模式不符。柔性模式下 `[ADR-???]` 槽位会被默默保留,没有阻止机制就无力。
- **分级模式**(路径白名单/黑名单决定是否严格):被排除,理由是配置负担大、边界定义主观,且本 skill 无法预知使用者项目结构。分级的真正实现形式在本决策中被编码为 `check_exemption` 的内部分支——由使用者在自己的项目里实现,而非 skill 强加。
- **cp 时默认不启用严格 hook,由 opt-in 指令开启**:被排除,理由是默认值决定实际行为。默认松散意味着大多数使用者永远不开启严格模式,spec-vc 的承诺变成空话。
