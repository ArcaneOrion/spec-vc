# ADR-022: 审计模式冻结对齐文档基线

- **Date**: 2026-05-25
- **Status**: Proposed
- **Deciders**: arcaneorion
- **Tags**: 

## Context

spec-vc review 当前把 staged diff 指纹写入 `.git/spec-vc-review.json`，commit-msg hook 再用该指纹确认审计证据对应当前提交内容。这个机制能发现 review 后代码发生变化，但不能发现 review 后 ADR、Plan、Spec 这些对齐文档发生变化。

下游项目实战中出现了这个风险：AI 在进入审计模式后仍可能改写对齐文档，让文档迁就代码，再用新的文档表面上证明代码“已经按规格审查”。此时问题不在模型是否诚实，而在审计系统没有冻结审计开始时的语义基线。

ADR-020 删除的是教 AI 如何思考的 reasoning scaffolding。本问题不同：它保护的是审计证据的时间边界，属于 operational harness。

## Decision

我们将在 `spec-vc review` 写入审计证据时记录文档基线，并在 commit-msg hook 中复算校验。

基线范围为当前 ADR、当前或最新关联 Plan、关联 Spec 的 `dev-doc.md` 与形式化文件 `contract.openapi.yaml`、`schema.json`、`behavior.feature`。基线以仓库相对路径、文件种类、存在性和 sha256 的形式写入 `review.json.document_baseline`。

当 `[ADR-NNN]` 提交触发 commit-msg hook 且未设置 `SPEC_VC_BYPASS` 时，hook 会在既有 anchor 与 mtime 校验之后比较文档基线。若基线文件集合、存在性或内容哈希发生变化，提交被阻塞，并提示重新运行 `spec-vc review`。

旧版 `review.json` 不含 `document_baseline` 时保持兼容，不因字段缺失而解析失败。

## Consequences

积极后果：

- 审计证据同时绑定代码差异和语义基线，review 后改写 ADR/Plan/Spec 会被发现。
- 机制只保护提交边界，不阻止用户修改文档；文档变化后重新 review 即可。
- 基线只记录路径和哈希，不把文档正文复制进 `.git/` 证据文件。

消极后果：

- 正当的 review 后文档修正会要求重新 review，多一次命令成本。
- Plan 选择规则必须确定，否则多个 plan 共存时可能出现误伤。

中性后果：

- `review.json` 数据形状新增可选字段，需要保持向后兼容。
- `[ADR-none]` 路径仍然直接放行，不纳入本次冻结机制。

## Alternatives Considered

1. **只靠 staged diff anchor**：被排除，因为 staged diff 不能表达审计时读到的 ADR/Plan/Spec 是否仍是同一版本。
2. **禁止 review 后编辑所有文档**：被排除，过于粗糙，会误伤 README、普通 docs 和合法规格修正。
3. **校验 `context_summary` 文本**：被排除，摘要是审查助手输出，不是规范化证据；比较摘要既不完整，也会把 carrots 变成新的 sticks。
4. **把 ADR/Plan/Spec 全文写入 review.json**：被排除，证据文件膨胀且可能泄露不必要内容；sha256 足以证明是否漂移。

## Implementation Plans

- **Plan**: doc/arch/plans/ADR-022-plan-001.md
- **Summary**: 实现审计模式文档基线冻结：新增 document_baseline 模块，spec-vc review 写入 ADR/Plan/关联 Spec 的路径、存在性与 sha256；commit-msg hook 在 [ADR-NNN] 路径下复算并阻塞 review 后的基线漂移。覆盖 review 后修改 ADR、Plan、关联 Spec 形式化文件三类回归，保留旧 review.json 兼容与 SPEC_VC_BYPASS 逃生口。
- **Pre-Validation**: 前置验证：Spec-022 已创建并 formalize all，spec check 返回“全部 12 个 Spec 就绪”。现状分析与 explorer 独立结论一致：当前 review.json 只含 staged diff anchor/context_summary，不含 ADR/Plan/Spec 文档基线；hooks.py:_check_review_record 仅校验 review.json 存在、anchor 匹配和 mtime 新鲜，不复算对齐文档哈希。因此 review 后修改 ADR/Plan/Spec 当前不会被 hook 识别。本轮修改前验证口径：先补失败测试复现该漏洞，再实现 document_baseline 写入与 hook 复算，使同一测试通过；最后运行 uv run pytest tests/python/ 全量回归。
- **Post-Validation**: 后置验证完成：先按 ADR-022 验证口径补失败测试，review 后修改 ADR/Plan 时当前 hook 放行，测试红灯复现漏洞；实现后同一组 document_baseline 测试全部通过（3 passed），覆盖 review 后修改 ADR、Plan、关联 Spec 形式化文件均被 commit-msg hook 阻塞。补充 test_review_writes_commit_msg 断言 review.json.document_baseline 写入 version/adr_token/ADR 文件基线，并跑兼容小回归（document_baseline + 旧 review.json/context_summary + bypass 相关 4 passed）。最终全量回归 uv run pytest tests/python/ 129 passed in 14.49s。Spec-022 已 formalize all，spec check 此前返回全部 12 个 Spec 就绪。
