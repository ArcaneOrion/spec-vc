# ADR-012: 门禁消息增强：失败时返回可执行指引而非仅阻塞

- **Date**: 2026-05-08
- **Status**: Proposed
- **Deciders**: arcaneorion
- **Tags**: commit-protocol, ux, ai-agent

## Context

ADR-011 简化了提交流程，将防护收拢到 commit-msg hook。但实际实施 ADR-011 时暴露了两个流程断点：

1. **改代码之前没有有效门禁**：`change validate --phase pre` 只检查全局 Spec 就绪状态，不检查 ADR 是否有关联 Spec、clarify 是否完成。AI 可以跳过 Spec 创作协议直接进入代码修改。
2. **门禁阻塞消息只有"不行"没有"怎么做"**：commit-msg hook 和 `validate --phase pre` 的错误消息是功能性描述（"未找到 subagent 审计记录"、"stage 为 clarify"），AI 读到后知道出了问题，但不知道正确的下一步该做什么。

这两个断点的根因相同：门禁设计假设使用者是人类（读完错误消息后自行判断下一步），但实际主要使用者是 AI Agent——它需要明确的可执行指令来形成正确行为意图。

## Decision

1. 所有门禁阻塞消息增加"下一步"指引，包含具体的 CLI 命令和操作步骤
2. 所有门禁阻塞消息末尾追加 `详细流程请查看 SKILL.md 检查正确流程`
3. `change validate --phase pre` 增加 clarify 完整性检查（stage 仍为 discover/clarify 时阻塞）和 ADR→Spec 关联检查（ADR 无关联 Spec 且变更涉及代码路径时提示）
4. `commit prepare` 的输出消息从"请运行 submit"改为"请审计后直接 git commit"并列举 hook 校验项

## Consequences

- 积极后果：AI Agent 遇到门禁阻塞时能自动形成正确的补救行动意图，减少流程跳步
- 消极后果：错误消息变长，人类在终端看到可能显得冗余
- 中性后果：SKILL.md 成为错误消息的权威引用点，其内容需要保持与代码一致

## Alternatives Considered

- **PreToolUse hook 在 AI 修改文件时拦截**：在 AI 调用 Write/Edit 工具修改 src/ 下文件时检查流程状态。更主动但侵入性太强，且需要精确判断哪些文件修改需要门禁。
- **在 _active.md 中增加阶段完成标记**：让门禁检查更有结构化依据。但不改变消息内容的话仍只解决"检查是否通过"不解决"告诉 AI 该怎么做"。
