# ADR-007: 为 commit token 门禁引入 emergency bypass 机制

- **Date**: 2026-04-30
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: hook, governance, emergency, bypass

## Context

ADR-006 通过 commit token 门禁，把"必须经过 spec-vc commit"做成了机制级强制。代价是 commit-msg hook 与 spec-vc binary 之间形成了运行时单点依赖——hook 调用 `~/.claude/skills/spec-vc/.venv/bin/spec-vc` 写入和校验 token，binary 损坏 / skill 路径变更 / venv 错乱时，整个仓库的 commit 通道立刻锁死。

这层风险在单人主项目自用阶段被掩盖（环境稳定），但在两个场景下成为阻断：

- **开源场景**：第一批使用者中必然有人把 skill 装到非默认路径、用不同 Python 版本、甚至误删 venv。token 门禁失效时没有预设逃生口，第一次踩坑等于弃用。
- **真实工程场景**：hotfix、CI 自动提交、bisect、断网调试等情况下，开发者需要绕过治理快速提交，没有逃生口意味着治理与生产力直接冲突。

设计张力在于：**逃生口本身是对 ADR-006 强制约束的削弱**。任何永久的、无门槛的开关都会让 token 门禁退化为"事实上的可选项"。需要在"可逃生"与"可追溯"之间找到平衡——逃生必须是显式、临时、留痕的。

## Decision

我们将引入 **环境变量级 emergency bypass + 审计日志** 机制：

1. `commit-msg` hook 在 token 校验之前检查环境变量 `SPEC_VC_BYPASS`。**非空字符串**触发 bypass，跳过 token 校验。空字符串或未设置时走原有 token 校验逻辑。
2. bypass 触发时，向 `.git/spec-vc-bypass.log` 追加一行审计记录，格式：`ISO 时间戳 | 原因 | commit subject`。日志文件在 `.git/` 下，不进入 git 追踪，与 `.git/spec-vc-commit-token` 同性质。
3. 日志写入采用 **fail-open** 策略：IO 异常（路径不可写、磁盘满等）时仅向 stderr 输出警告，hook 仍放行 commit。逃生口不引入新的阻塞点。
4. bypass 仅跳过 token 校验。**ADR 引用校验（`[ADR-NNN]` / `[ADR-none]`）和豁免规则（exemption_allows）照常生效**——bypass 不是治理失能开关，仅解除 ADR-006 引入的通路约束。
5. token 门禁阻塞错误信息显式列出 `SPEC_VC_BYPASS=<原因> git commit ...` 用法，降低用户拼错环境变量名的概率。
6. README 增加"紧急绕过 / Troubleshooting"段，说明环境变量用法、原因字段建议（`hotfix` / `ci` / `bisect` / `migration` / `repair`），以及机制本身崩溃时的最终兜底——`chmod -x .git/hooks/commit-msg` 让 git 原生跳过 hook。

## Consequences

**积极**:
- 单点故障不再锁死仓库——spec-vc binary 损坏 / skill 路径变更 / venv 错乱时仍有显式逃生路径
- 开源初期使用者的"第一次踩坑"不再等于弃用——bypass 用法在 hook 阻塞错误信息里直接打印
- 审计日志保留 bypass 历史，事后可识别滥用模式（如全是 `1` / `ok` 的原因字段说明用户在习惯性绕过），为未来加强治理留出反馈通道
- 对 ADR-006 token 门禁的削弱被限定在最小范围——仅跳过 token，不削弱 ADR 引用、豁免规则、Spec 就绪检测等其他治理层

**消极**:
- bypass 不可避免地存在被滥用的可能——用户填 `SPEC_VC_BYPASS=1` 之类无意义原因即可绕过，机制层无法阻止形式合规的虚假填写
- 审计日志位于 `.git/`，不跨克隆同步——多人协作场景下 bypass 历史不可见，他人无法察觉滥用
- 引入新的状态文件 `.git/spec-vc-bypass.log`，需在卸载 / 仓库清理流程中考虑
- 错误信息中明示 `SPEC_VC_BYPASS` 用法，本身降低 bypass 使用门槛，与"逃生口应当难以被随手触发"的反向直觉冲突——但相比拼错变量名导致死锁，明示是务实选择

**中性**:
- 原因字段无格式约束（仅强制非空）——保留事后审计弹性，代价是可能堆积无意义字符串
- 不接管"项目级永久禁用 token 门禁"需求——该需求由 git 原生 `core.hooksPath` 或手工 `chmod -x` 覆盖，不由 spec-vc 提供开关，避免治理意图被稀释

## Alternatives Considered

- **双触发通道（环境变量 + `git -c spec-vc.bypass=<reason>`）**：被排除——git config 通路确实更可靠传递，但单人 + 开源初期场景下环境变量已足够；hook 错误信息显式打印用法可覆盖拼错风险。引入双通道意味着两套校验逻辑、两套测试，与"小修小改"原则冲突。等真实使用反馈"环境变量传不进去"再加。

- **commit message 追加 `[BYPASS:<reason>]` 标记**：被排除——优点是审计证据进入版本控制，跨克隆可见；缺点是需要在 commit-msg / prepare-commit-msg 阶段联动改写 message 文件，复杂度显著高于 append 一行日志。单人项目场景下 `.git/spec-vc-bypass.log` 已覆盖事后追溯需求；开源后若有团队协作场景报告"需要跨机器审计"再加。

- **配置项永久关闭 token 门禁（`.spec-vc.toml` 添加 `[token] enabled = false`）**：被排除——这本质是"反正能关"的配置开关，会让 token 门禁从一开始就退化为可选项，违反 ADR-006 决策的强制性。emergency 应当是临时的、显式的、留痕的，不是配置默认值的一部分。

- **强制原因白名单（`hotfix` / `ci` / `bisect` 等枚举）**：被排除——白名单看似提升追溯质量，但用户找不到合适词时要么放弃 bypass（卡死，回到锁死状态），要么乱填一个白名单内的词（更难识别滥用）。审计目标不是"防止填垃圾"，而是"被滥用后能识别滥用模式"——非空字符串 + 推荐词 hint 已达成此目标。

- **CLI flag（如 `git --bypass-spec-vc commit`）**：被排除——git 不接受未识别的 flag，需要 wrapper 命令，引入新的命令面与心智成本。环境变量是 git hook 场景下的天然机制，不需要重新发明入口。

## Implementation Plans

- **Plan**: doc/arch/plans/ADR-007-plan-001.md
- **Summary**: ADR-007 成功为 ADR-006 token 门禁引入最简 emergency bypass 机制：在 hooks.py:run_commit_msg 中增加环境变量 SPEC_VC_BYPASS 检查分支——非空时跳过 token 校验并写审计日志到 .git/spec-vc-bypass.log（fail-open，写入失败仅 stderr 警告仍放行）；ADR 引用与豁免规则照常生效，仅跳过 token 校验。同步更新 commit.py 错误提示信息显式列出 bypass 用法；README 新增'紧急绕过'段说明环境变量用法和 chmod -x .git/hooks/commit-msg 最终兜底方案。新增 3 个 pytest 测试覆盖 bypass 触发、空字符串回退 token 校验、日志写入失败仍放行三种场景。全量 94 测试通过。本次变更也是 spec-vc 首次完整 dogfooding——从 ADR-007 决策→Spec-002 行为规则→pre/post validation→close 回填全链路闭环。过程暴露并修复了 _sections.py replace_section 在 body 以数字开头时触发 invalid group reference 的 bug（已在 [ADR-005] 下独立修复）。
- **Pre-Validation**: 修改前状态：1) hooks.py:run_commit_msg 不检查 SPEC_VC_BYPASS 环境变量，token 校验无逃生口——spec-vc binary 损坏 / skill 路径变更会锁死整个仓库 commit；2) 阻塞错误信息仅提示'请通过 spec-vc commit 流程提交代码'，未列出 bypass 用法，用户无法发现逃生路径；3) 全量 pytest 基线 91 个测试通过（90 历史 + 1 个 _sections bug fix）；4) .git/spec-vc-bypass.log 不存在；5) README 无紧急绕过 / Troubleshooting 段。已通过 spec check 确认 Spec-002 全部就绪可作为审计依据。
- **Post-Validation**: 修改后验证：1) 94/94 pytest 全部通过（基线 91 + 新增 3 个 bypass 测试）；2) SPEC_VC_BYPASS=hotfix hook commit-msg 无 token 时放行且 .git/spec-vc-bypass.log 新增一行含 'hotfix' 和 commit subject；3) SPEC_VC_BYPASS 未设置时 git commit 仍被 token 门禁阻塞；4) 阻塞错误信息显式打印 SPEC_VC_BYPASS 用法；5) 把 .git/spec-vc-bypass.log 做成目录后 SPEC_VC_BYPASS=repair 仍能 commit（fail-open 验证通过）；6) README 末尾新增'紧急绕过'段，说明环境变量用法、原因字段建议和 chmod -x 兜底方案。
