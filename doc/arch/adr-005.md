# ADR-005: 引入 Spec 结构化规格层与 subagent 审计协议

- **Date**: 2026-04-26
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: spec, audit, layer-3

## Context

ADR-000 建立了"决策版本控制"(Layer 2),解决了"为什么这么做"的记录问题。但 Agent 辅助开发环境下还有第二个裂缝:Agent 按自然语言 ADR 写代码时,对"应该做什么"的理解存在模糊空间——同样一份 ADR,不同时间点可能被翻译成不同的接口形态。

Layer 3(规格版本控制)需要解决的是"应该做什么"的版本化与可验证。但面临现实约束:
- 纯形式化验证(Lean 4 / TLA+)在当前阶段编写成本过高,世界上没有成熟的通用方案
- 需要在"可验证"和"可落地"之间找到务实的中间路径

核心洞察:三层文档形成渐进校对流程。
- 校对点 1(ADR):决策是否合理?人工审阅
- 校对点 2(dev-doc):翻译是否准确?人工对照 ADR
- 校对点 3(Spec 形式化文件):编码是否正确?机械验证 + subagent 审计

## Decision

我们将:

1. 引入三层文档体系,作为变更治理的标准流程:
   ```
   ADR Plan → 结构化开发文档(dev-doc.md) → Spec 形式化文件(.yaml/.json/.feature) → Code
   ```
2. Spec 文件存放在 `doc/arch/specs/{NNN}/` 子目录中,与 ADR 同层但独立管理
3. 形式化文件按类型分拆:接口契约用 OpenAPI (.yaml)、数据形状用 JSON Schema (.json)、行为规则用 Gherkin (.feature)
4. 引入 `spec-vc commit` 双 subagent 验证协议:
   - Subagent A(审计):对照 Spec + dev-doc + git diff,检查代码实现是否符合规格
   - Subagent B(测试):仅基于 Spec 形式化文件生成并执行测试,不阅读代码
5. 验证结果决定 commit 是否放行:双通过 → 自动 git commit;任一失败 → 清理测试 → 回到对话修复
6. clarify 流程从"机械追问 6 字段"改为"自然讨论 + AI 内部 6 方面 checklist"

## Consequences

**积极**:
- Spec 文件进入版本控制,与 ADR 双向锚定,变更链完整可追溯
- 双 subagent 从正反方向验证代码,审计看代码,测试不看代码,互补减少漏检
- dev-doc.md 保留结构化文档的阅读价值,不因形式化而牺牲可读性
- clarify 流程改为自然对话后,降低机械摩擦,提高对齐效率

**消极**:
- 三层文档增加了维护负担,每次变更需要更新 2-3 个文件
- subagent 审计依赖 LLM 能力,不是纯机械过程,存在漏检可能
- Spec 形式化文件的编写成本高于自然语言 ADR

**中性**:
- 测试文件通过后保留在 `specs/*/tests/` 中作为回归基准,失败时删除
- subagent 审计协议的提示词模板固化在 commit.py 中,可按项目定制

## Alternatives Considered

- **跳过 dev-doc 层,直接从 ADR 生成 Spec**: 被排除——dev-doc 在"人可读"和"AI 可解析"之间提供了关键翻译层,且程序员普遍能看懂
- **纯机械验证(OpenAPI validator / spectral / Cucumber)**: 未排除,但当前阶段 subagent 审计是务实桥梁——未来的形式化拓展可以基于 subagent 架构
- **单 subagent 统一审计+测试**: 被排除——输入源隔离(审计看代码,测试不看代码)是两个独立验证方向的保证
