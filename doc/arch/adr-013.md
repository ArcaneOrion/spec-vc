# ADR-013: hook 校验链补完：adr_id 路由与 session log 时间戳新鲜度

- **Date**: 2026-05-08
- **Status**: Proposed
- **Deciders**: arcaneorion
- **Tags**: commit-protocol, hook, audit-integrity

## Context

ADR-011/012 收尾时暴露两个 hook 校验链补完点：

1. **`hooks._load_active_stage(adr_dir, adr_id)` 函数体未使用 `adr_id` 参数**：永远读 `_active.md` 的 stage，与传入 ADR 无关。当 commit subject 引用 `[ADR-X]` 而 active 是 ADR-Y 时，校验对象错位。当前因"单 active change"约束未爆发，但是隐性 bug，未来扩展（追加 commit、并行 ADR）时会失效。pyright 已标 `adr_id is not accessed`。

2. **PostToolUse hook 仅以"session log 非空"作审计证据**：今天上游 Agent API 500 失败时，PostToolUse hook 仍写入 description 为空的日志行（5 行中有 3 行空 description），commit-msg hook 形式上仍通过。这回到 ADR-008/011 想堵的"仪式性防护"——审计是否真实发生不被检验。

两点都属于 ADR-011/012 设计意图的"漏检"，不是新功能。

## Decision

1. `_load_active_stage` 重写为 `_load_stage_for_adr(adr_dir, adr_id)`：active 匹配则用 active.stage；不匹配则从 `plans/ADR-{adr_id}-plan-*.md`（取编号最大的）读 `- **Stage**:` 字段；该 ADR 无 plan 文件 → 返回 None（视为流程已结束，不阻塞）。

2. `commit.py` 新增 `check_session_log_freshness(repo_root)`：比较 `.git/spec-vc-commit-msg` mtime 与 session log 末行时间戳，要求末行时间 > commit-msg mtime（证明审计在写入 commit-msg 之后发生）。`commit-msg` 文件不存在（用户未走 `prepare` 直接 `git commit`）则跳过该项检查（保留旁路）。

3. `run_post_tool_use` 当 description 为空时不写日志（避免空行污染 session log）。这是对 ADR-009"全量记录"意图的收紧——只记录有 description 的 Agent 调用。

## Consequences

- 积极后果：(1) 解除 hook 校验链对"单 active change"假设的隐性依赖，未来可平稳扩展并行/历史 ADR 提交；(2) freshness 检查闭合"仪式性 vs 实质审计"的最后缺口——审计必须真实发生在 commit-msg 写入之后；(3) 减少空日志行污染。
- 消极后果：freshness 检查可能误伤"prepare 后等待数小时再 commit 但中途无新 Agent 调用"的场景——但这种情况说明审计陈旧，阻塞合理；用户可用 `SPEC_VC_BYPASS` 旁路。
- 中性后果：session log 不再代表"全部 Agent 调用"，只代表"有效 Agent 调用"——下游若有依赖 session log 做事件审计，需要重新评估口径。

## Alternatives Considered

- **PostToolUse 检测 Agent 调用是否成功**：通过额外 hook 参数判断 success/failure，仅记录成功调用。技术上可行但需修改 hook 注册参数 + Claude Code harness 配合，超出本次范围。改为"description 非空"作为代理检查，简单且足够。
- **保持 `_load_active_stage` 签名不变，仅删 adr_id 参数**：能消除 lint warning 但不解决"未来支持多 ADR 时如何按 adr_id 路由"的设计空白。重写更面向未来。
- **freshness 检查比较 commit 时刻而非 commit-msg mtime**：更严格，但 commit-msg hook 在 git commit 进程内运行，无法可靠取"commit 起始时间"。用 commit-msg mtime 作锚点更稳。
