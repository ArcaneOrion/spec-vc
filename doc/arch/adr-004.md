# ADR-004: init命令增加uv环境安装步骤

- **Date**: 2026-04-26
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: 

## Context

spec-vc 作为 Python CLI 工具,依赖虚拟环境管理。工作环境是 NixOS,Python 管理用 uv。

问题:skill 加载后首次调用 spec-vc CLI 时,若未预先 `uv sync`,会直接报 `zsh: command not found: spec-vc`。SKILL.md 中部分 CLI 调用写了裸 `spec-vc`,假设环境已激活——这个假设在 Claude Code 的 skill 环境中不成立。

力量:
- Claude Code skill 环境不会自动激活项目 venv
- uv run 是 uv 提供的零配置入口,不依赖 venv 激活状态
- SKILL.md 的启动协议应该对环境假设最保守

## Decision

我们将:

1. `init` 命令新增 `uv sync` 步骤,在创建 ADR 目录和 hooks 之前先确保环境可用
2. SKILL.md 启动协议中所有 CLI 调用统一使用 `uv run spec-vc ...` 前缀,不假设 venv 已激活
3. `pyproject.toml` 添加 `[build-system]` 声明,使 `uv sync` 能安装 spec-vc entry point

## Consequences

**积极**:
- 首次加载 skill 后不会出现 `command not found`
- `uv run` 作为统一入口,无需用户记忆"要不要先激活 venv"
- CLI 入口确定性提高,降低启动摩擦

**消极**:
- `init` 命令执行时间增加(uv sync 需要网络)
- `uv run` 比直接 `spec-vc` 多 6 个字符

**中性**:
- 构建系统声明向 PEP 517 靠拢,未来如需发布到 PyPI 可平滑过渡

## Alternatives Considered

- **假设 venv 已激活,在 SKILL.md 里写"请确认已运行 uv sync"**: 被排除——依赖人记忆,启动摩擦仍然存在,且与"让机器人更可靠"的设计哲学冲突
- **只在 README 里加提示,不改代码**: 被排除——提示文档的发现概率远低于 CLI 报错时的直接反馈
- **使用 pipx 安装**: 被排除——项目没有发布到 PyPI,且 pipx 在 NixOS 上有额外配置负担

## Implementation Plans

- **Plan**: doc/arch/plans/ADR-004-plan-001.md
- **Summary**: init 新增 uv sync 环境安装步骤确保 CLI 可用；SKILL.md 统一 uv run 前缀；pyproject.toml 添加 build-system 让 uv sync 安装 entry point
- **Pre-Validation**: 现有 init 命令不包含环境安装步骤；SKILL.md 仍有裸 spec-vc 调用；需确认 uv run 在无 venv 激活时可用
