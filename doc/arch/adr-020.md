# ADR-020: spec-vc 做减法：删除 reasoning scaffolding，保留 operational harness

- **Date**: 2026-05-24
- **Status**: Proposed
- **Deciders**: arcaneorion
- **Tags**: meta, refactor, harness-design

## Context

### 初心重申

spec-vc 的设计初心：**让 AI 写代码与人类语义对齐**，融合代码审查与决策记录。

拆解：
- **语义载体**：人类用 ADR 表达"为什么这么做"、用 Spec（OpenAPI/JSON Schema/Gherkin）表达"做成什么样"
- **AI 消费**：AI 在写代码时能读到并 reference 这些语义
- **机制**：审查（保证 AI 没偏离语义）+ 决策记录（追溯可读）

### 项目演化轨迹与认知偏差

ADR-000 ~ ADR-019 的实际演化呈现出一个偏离初心的倾向：

| ADR 角色 | 占比 | 性质 |
|---|---|---|
| **语义载体**（ADR/Spec 创作流程、形式化文件） | 少数 | 服务初心 |
| **降低 AI 消费成本**（ADR-012 错误可执行、ADR-019 review 助手主动呈现） | 少数 | 服务初心 |
| **逼 AI 真去消费语义载体 / 监控 AI 行为**（ADR-006~011/013~018 多数） | 多数 | 辅助手段被当成目的 |

具体可见的失衡迹象：
- commit-msg hook 从 ADR-006 的单一 token 校验演化到 ADR-018 的 6 步校验链
- ADR-017 的 anchor binding 后即被 ADR-018 的 review.json 取代（净增机制 + 净删机制）
- ADR-010 一次性删除 1374 行（ADR-005~008 累积的 reasoning scaffolding 自我否定）
- ADR-017/018/019 出现"sticks→carrots""senior dev workflow""设计哲学转向"等抽象叙事，与实际代码改动密度不匹配

### 业界证据

2024-2026 的工业界与学术界对 agent 设计形成新共识：

**1. Claude Code v2.1.88 (512K LOC) 逆向工程**（VILA-Lab, arXiv 2026）发现：

> 1.6% 是 AI 决策逻辑，98.4% 是确定性 operational harness（权限、上下文压缩、工具路由、恢复）。设计哲学：minimal scaffolding, maximal operational harness。

> "An agent is a model. Not a framework. Not a prompt chain. The model is smarter than your prompt. Design the environment, not the behavior."

**2. Anthropic 官方 Building Effective Agents**（Schluntz & Zhang, 2024-12）：

> "The most successful implementations weren't using complex frameworks or specialized libraries. Instead, they were building with simple, composable patterns."

**3. Anthropic Claude 4.x prompt engineering guide**：

> "If you've added scaffolding to force interim status messages ('After every 3 tool calls, summarize progress'), try removing it."

明确建议：Claude 4.x 字面执行而非补全意图，旧 scaffolding 反成噪声。

**4. Cognition "Don't Build Multi-Agents"** (Walden Yan, 2025-06)：单线程线性 agent + 强 context 管理优于复杂编排。

### 关键判别法（本 ADR 的核心工具）

把任意机制按 VILA-Lab 框架二分：

| 类型 | 留 | 删 |
|---|---|---|
| **Operational harness** | 权限/审计/工具沙箱/上下文管理/错误恢复/追溯证据 | – |
| **Reasoning scaffolding** | – | 教模型怎么思考/显式 state graph/强制 reasoning trace/prompt 模板里的步骤指引 |

Operational harness 在威胁模型存在时不可省（合规、审计、关键基础设施场景）。Reasoning scaffolding 在模型能力上来后变成负贡献。

### 触发本 ADR 的具体张力

- 自我审视 + 外部 AI 评价显示 spec-vc 当前向 reasoning scaffolding 一侧倾斜
- ADR-019 自举验证时 ruff 报告暴露真实问题（adr.py unused import），证明 "design the environment" 路径在 spec-vc 上有效
- 用户基于大量 AI 使用经验的判断："AI 不审查是因为成本高"——这进一步支持"降低消费成本 > 提高违规成本"

## Decision

**1. 删除 reasoning scaffolding 类机制**：

| 机制 | 所在 ADR | 性质 | 处置 |
|---|---|---|---|
| commit-msg hook plan stage 校验（≥ implement-ready） | ADR-011 | 教 AI 走流程 | **删** |
| commit-msg hook simple 模式 note 含 anchor 子串校验 | ADR-018 | 教 AI 怎么"诚实" | **删** |
| commit-msg hook `[ADR-none]` 量化判定（lightweight.py 整体） | ADR-018 | 教 AI 什么算轻量 | **删** |
| `require_user_verified` 配置项 | ADR-018 | 强制工作流 | **删配置项**（review.json.verified 字段保留作记录） |

**2. 保留 operational harness 类机制**：

| 机制 | 性质 |
|---|---|
| SPEC_VC_BYPASS → audit log | 审计追溯 |
| commit-msg ADR 引用格式校验 | 保证 commit 可追溯到决策 |
| commit-msg Spec 完整性校验（dev-doc + 形式化文件就绪） | 保证语义载体真实存在 |
| commit-msg review.json 存在 + anchor 匹配 + mtime 新鲜 | 防止 audit 与 staged 内容失配 |
| BlockingError 结构化输出 | 错误恢复路径 |
| ADR-019 review 助手报告 | environment design 的核心实现 |

减法后 commit-msg hook 校验链从 6 步压到 4 步。

**3. 建立 ADR 写作规范**（写入 `CLAUDE.md`，强制约束）：

- 每条 ADR 必须自包含可读（不依赖前置 ADR 才能理解）
- Plan summary 必须含 ≥ 1 个具体 file:line 或 commit hash 引用
- 禁用宣示句式：「设计哲学转向」「心智模型」「X 取代 Y」「sticks/carrots」类
- 哲学讨论 ≤ 1 段；超过的写到独立回顾文章
- "AI 行为假设"必须有 bypass log 或测试数据支撑，否则不写

**4. 不删的明确边界**（防止过度减法）：

- Spec 形式化契约系统（这是 environment 的核心）
- ADR/Spec 创作协议（语义载体的生产线）
- review 助手主动输出（ADR-019 的核心 carrots）

## Consequences

**积极**：
- commit-msg hook 校验链 6→4 步，逻辑减半，测试相应减少 ~15-20 项
- 跨项目部署阻力下降（少 1 个配置项 / 少 1 个量化阈值需要调）
- ADR-020 后写作规范硬约束防止重蹈"哲学叙事覆盖工程"覆辙
- 接近初心："设计环境，不设计行为"

**消极**：
- 失去 plan stage 校验后，AI 可能在 stage=clarify 时提前 commit；缓解：靠 Spec 完整性校验兜底（spec 没就绪就阻塞）
- 失去 [ADR-none] 量化判定后，AI 用 [ADR-none] 提交大改动的概率上升；缓解：信任 AI 判断 + 事后通过 git log 审计可追溯
- 删除 simple 模式 note anchor 校验后，simple 模式回到纯 honor system；缓解：review 助手报告已经把 anchor 主动展示，"假装审查"成本仍存在

**中性**：
- 部分 ADR-006 ~ ADR-018 的实施记录被 partial supersede；不删除历史 ADR 文件
- 用户从 .spec-vc.toml 自定义 [lightweight] / require_user_verified 的能力消失（这些都不再被读）

## Alternatives Considered

**A. 不做减法，继续加机制（默认惯性）**
- 排除理由：已观察到的失衡——19 条 ADR 中只有少数服务初心，多数在做次级问题。继续加只会加速偏离

**B. 全删 commit-msg hook 校验链，只留 ADR/Spec 创作流程**
- 排除理由：丢掉了 operational harness 价值。VILA-Lab 证据显示 98.4% 的可靠性来自确定性 harness，不是 0。BYPASS audit、ADR 引用、Spec 完整性、review.json anchor 这些是审计可追溯的基石

**C. 冻结 commit-msg hook 半年，只修 bug 不加新校验**（前一个外部 AI 评价提出）
- 部分采纳：本 ADR 的写作规范包含"删减/简化"导向，相当于冻结"加法思考方式"
- 但冻结本身不够——它不解决"已有的 reasoning scaffolding"，只是冻结增量。本 ADR 选择主动删而非被动冻结

**D. 做减法但保留所有现有测试（标记 skip）**
- 排除理由：保留 skip 的测试是死代码，违反 spec-vc 自身"don't add features beyond what the task requires"原则。删机制就删配套测试

**E. 等跨项目实战数据再决定（让"AI 不审查是因为成本高"的假设先在外部项目验证）**
- 部分采纳：本 ADR 不阻塞跨项目验证，验证结果可作为 ADR-021 的输入
- 但当前已有足够证据（VILA-Lab + Anthropic 官方 + 自举观察）支持做减法，不需要再等
