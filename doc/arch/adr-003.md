# ADR-003: skill 扁平化 + 命令前缀统一为 /spec-vc adr-*

- **Date**: 2026-04-20
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: naming, architecture, refactor

## Context

v0.1 初期本仓库采用**嵌套结构**:

```
spec-vc/
├── README.md
└── decision-vc/       ← skill 本体(name: decision-vc)
    ├── SKILL.md
    └── ...
```

命名意图:`spec-vc` 是三层框架的**伞仓库**,`decision-vc` 是其中**第一层**(决策版本控制)的独立 skill;未来加第二层实现时,会新增 `spec-vc-skill/` 等兄弟目录。

初次实施后识别出几个问题:

1. **概念切分错位**:三层版本控制的真实关系是**同一个工具的三个能力层**(ADR / Spec / 强形式化规格),是**逐步演化**而非"三个独立 skill"。合适的命名应反映演化,而不是人为拆成多个 skill。
2. **未来命名冲突**:仓库名 `spec-vc` 与 v0.3 计划中的"规格层 skill"撞名,只能给 skill 取别扭的 `spec-vc-skill/`。
3. **一层冗余目录**:skill 本身被嵌套在仓库根的子目录里,用户 clone 后不能直接加载,还要"进一层"。
4. **命令前缀与 skill name 绑定**:Claude Code 的 skill 机制通常按 skill name 路由命令,`/decision-vc *` 意味着升级到规格层时必须更名,造成历史引用断层。

决策时机约束:只有 3 条 ADR、3 条 commit,此时收敛命名的代价最低;越往后拖,引用点越多,迁移越贵。

## Decision

我们将:

1. **扁平化**:把原 `decision-vc/` 下的 `SKILL.md / commands / hooks / scripts / templates / tests` 全部上提到仓库根,仓库本身即为 skill。
2. **SKILL name = `spec-vc`**:架构命名反映**终极愿景**(三层版本控制),而不是"当前实现了哪一层"。当前用户看到 `spec-vc` skill 会期待规格功能——在 README 最前面用**实现进度表**明确声明"当前只开发到 Layer 2(决策版本控制),Layer 3 规格版本控制在设计开发中"。
3. **命令前缀统一为 `/spec-vc`**,层间用**子命令前缀**区分:
   - Layer 2(当前):`/spec-vc adr-init`、`adr-new`、`adr-link`、`adr-status`、`adr-list`、`adr-upgrade`
   - Layer 3(未来):`/spec-vc spec-init`、`spec-new`、`spec-validate` 等
4. **历史 ADR 文本同步**:ADR-000/001/002 的文本中所有 `/decision-vc *` 和 `decision-vc skill` 被同步替换为新命名,作为本 ADR-003 的 commit 一并提交。ADR 内容本身不变,只是表述更新——不触发状态机迁移。

## Consequences

**积极**:
- 升级到 Layer 3 时无需重命名 skill,历史引用永不断链
- 仓库结构扁平,clone 后可直接加载,减少一层认知负担
- 命令命名空间有**结构表达力**(`adr-*` vs `spec-*`),用户从命令名就能识别所属层
- 架构命名承诺了"这是连续演化的单一工具",与路线图一致

**消极**:
- SKILL name 超出当前能力范围——`spec-vc` 字面上暗示"规格版本控制",但当前只能做决策版本控制,需要 README 的实现进度表持续负担这一解释义务
- 命令变长(`adr-init` 比 `init` 多 4 字符),日常调用成本略增
- 对已内化旧命名的使用者(主要是 arcaneorion 自己),需要一次心智切换

**中性**:
- 本次迁移发现并顺手修了 `scripts/new-adr.sh` 的 sed 分隔符 bug(title 含 `/` 会破坏替换):改用 awk 字面量替换,绕开 sed 对特殊字符的敏感性。该修复随本 ADR 的 commit 一并提交。
- ADR-003 自身的编号经历了一次"占用→释放→复用":sed bug 留下空文件 adr-003.md,修好后脚本从 004 起编,随即重命名回 003。这个过程是对"ADR 编号单调递增但可跳"不变量的现实检验——我们选择**复用**(3 从未真正被占用,只是被空文件短暂占位),而非**跳过**(留下永久空位),以保持编号连续性。
- ADR-003 的首次 commit(e4b62ab)因 Write 工具的"未先 Read"约束被写入了骨架而非完整内容;随后补了一个修正 commit 补全 Context/Decision/Consequences——这个过程本身被 `[ADR-003]` 锚定,后续 `git log --grep='\[ADR-003\]'` 会看到两条,符合"一 ADR 可对多 commit"的设计。

## Alternatives Considered

- **保留嵌套 `spec-vc/decision-vc/` + 未来加 `spec-vc/spec-vc-skill/`**:被排除,理由是命名冲突(伞仓库与子 skill 同名)、加载入口不够直观、skill 切分与实际"单一工具演化"的现实不符。
- **SKILL name = `decision-vc`,未来再改名为 `spec-vc`**:被排除,理由是改名会让历史 ADR 引用、commit message 规范、使用者习惯等多处断层;今天一次性迁移比未来分两次迁移更省总成本。
- **命令前缀分裂**(`/decision-vc *` 保留 + 未来加 `/spec-vc *`):被排除,理由是一个 skill 两个命令前缀不符合 Claude Code 的 skill 约定,且用户心智模型会分裂。
- **给命令加数字层标识**(如 `/spec-vc L2-init`、`/spec-vc L3-new`):被排除,理由是层数概念对使用者不透明——`adr-*` vs `spec-*` 的语义前缀比 `L2-*` vs `L3-*` 更自解释。
