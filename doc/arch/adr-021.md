# ADR-021: 修复 hook/venv 入口确定性，兼容 Codex

- **Date**: 2026-05-24
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: hooks, install, portability

## Context

spec-vc 的 commit-msg / prepare-commit-msg hook 由 `src/spec_vc/cli.py:_install_hook()` 写入目标仓库的 `.git/hooks/`，模板见 `hooks/commit-msg` 与 `hooks/prepare-commit-msg`，关键占位符是 `{{SPEC_VC_BIN}}`。在 0.3.0 之前，模板里先尝试 `command -v spec-vc`（PATH 优先），再 fallback 到 init 注入的路径。

实测时出现两个失败模式：

1. **PATH 抢占**：用户项目（例如 quant 仓库）激活了自己的 `.venv`，PATH 中 `spec-vc` 指向项目 venv 里的同名旧版本入口，hook 调用后报 `ModuleNotFoundError: spec_vc`。
2. **`~` 不展开**：cli.py 之前用 `Path.home() / ".claude/skills/spec-vc/.venv/bin/spec-vc"` 经 `.resolve()` 后写入，导致 `.claude/settings.json` 里出现机器绝对路径（如 `/home/arcaneorion/...`），跨开发者复制配置时污染；改成 `~` 字面值后又踩到第二个坑：bash 变量 `SPEC_VC_BIN="~/.claude/..."` 里的 `~` 在 `[[ -x "$SPEC_VC_BIN" ]]` 上下文中不会展开，hook 永远 fallback 到 PATH。

约束：

- spec-vc 必须在 Claude Code / Codex / 其他 CLI agent 用普通 `git commit` 触发时都正确命中 skill venv 入口（不能依赖 Claude 专属环境）。
- 入口选择必须确定，不能被项目 venv 或 PATH 顺序影响。
- 写入 `.claude/settings.json` 与 hook 的路径必须可移植，不能把开发者主目录硬编码。
- 不改变 ADR-018/019/020 的 review.json、Spec 完整性、anchor、mtime 校验语义。

## Decision

1. `hooks/commit-msg`、`hooks/prepare-commit-msg` 调整为 **skill venv 优先、PATH 兜底** 顺序：先 `[[ -x "$SPEC_VC_BIN" ]]` 调用 init 注入的字面路径，仅在该路径不可执行时才 `command -v spec-vc` fallback（见 `hooks/commit-msg:6-15`、`hooks/prepare-commit-msg:6-15`）。
2. `src/spec_vc/cli.py:_install_hook()`（`cli.py:65-70`）与 `.claude/settings.json` 写入器（`cli.py:99`）统一使用字面字符串 `"$HOME/.claude/skills/spec-vc/.venv/bin/spec-vc"`，依赖 bash 在 `exec` / `command` 执行时展开 `$HOME`；不再使用 `~` 字面（bash 变量不展开）、也不再使用 `Path.home().resolve()`（跨开发者污染）。
3. `spec-vc init` 仍然幂等覆盖目标仓库的 hook 与 settings.json，因此存量旧 hook 重新 init 即可治愈。
4. 新增 `tests/python/test_cli.py` 覆盖：init 写入的 hook 模板里 `SPEC_VC_BIN` 是字面 `$HOME` 路径、是可执行优先项；settings.json 不含开发者绝对路径。

## Consequences

**积极**：
- Codex / 其他 CLI agent 走普通 `git commit` 时 hook 行为与 Claude Code 一致，不再被项目 venv 抢占。
- `.claude/settings.json` 可被跨开发者复制，hook 模板里 `$HOME` 在每个用户的 shell 下自然展开。
- 旧 hook 通过 `spec-vc init` 一键治愈，不需要手动改 `.git/hooks/`。

**消极**：
- hook 现在硬假设 `$HOME/.claude/skills/spec-vc/.venv/bin/spec-vc` 是 deterministic 入口；如果用户把 skill 安装到非默认路径（如 `XDG_DATA_HOME` 自定义），需要重新 init 或手动调整。
- shell 必须支持 `$HOME` 展开（POSIX shell 都满足）；非 bash 极简环境下需要额外验证。

**中性**：
- 引入"skill venv 是 hook 的唯一可信入口"这一隐性合同，未来 skill venv 重命名或迁移需要同步更新 hook 模板和 cli.py。

## Alternatives Considered

1. **完全依赖 PATH（保持 0.2.x 行为）**：被排除，因为项目 venv 会抢占 PATH。
2. **在 hook 里手动展开 `~`（如 `${SPEC_VC_BIN/#\~/$HOME}`）**：被排除，bash 字符串替换语法可读性差且 portability 不如直接写 `$HOME`。
3. **用 `.resolve()` 写绝对路径**：在 394d451 已尝试并回滚，问题是 `.claude/settings.json` 跨开发者复制时出现机器绝对路径。
4. **要求用户在 shell rc 中导出 `SPEC_VC_BIN` 环境变量，hook 不写默认值**：被排除，违反"init 完即可用"的零配置承诺。
