# ADR-006: 引入 commit token 门禁机制，强制通过 spec-vc commit 提交

- **Date**: 2026-04-27
- **Status**: Accepted
- **Deciders**: arcaneorion
- **Tags**: hook, governance, enforcement

## Context

ADR-001 通过 `commit-msg` hook 强制 commit subject 包含 `[ADR-NNN]` 或 `[ADR-none]`，把 ADR 引用从"自觉"变成"机制"。但这层机制只校验 **message 格式**，不校验**提交通路**。

ADR-005 在此之上引入 Spec 形式化层与双 subagent 审计协议，期望 AI 在每次 commit 前通过 `spec-vc commit` 完成 Spec 就绪检测、manifest 收集、双 subagent 验证。但这条流程是**AI 自觉调用**——只要 commit message 带合法的 `[ADR-NNN]` 或 `[ADR-none]`，AI 直接执行 `git commit` 同样能通过 hook。

实际观察到的偏差：AI 绕过 `spec-vc commit` 直接 `git commit`，验证协议被静默架空。问题不在 AI 是否守约，而在**约束机制本身没有覆盖提交通路**——ADR-005 的整套协议建立在一个未被强制的前提之上。

## Decision

我们将引入 **commit token 门禁机制**，把"必须经过 spec-vc commit"从约定升级为强制：

1. `spec-vc commit` 在执行末尾向 `.git/spec-vc-commit-token` 写入一次性 token，含 300 秒过期时间
2. `commit-msg` hook 在校验 `[ADR-NNN]` 格式**之前**先校验 token 存在且未过期，缺失或过期则阻塞 commit 并提示"请通过 spec-vc commit 流程提交代码"
3. token 一次性消费——hook 校验通过后立即删除，下次 commit 必须重新经过 `spec-vc commit`
4. token 文件位于 `.git/`，不进入 git 追踪，是纯运行态状态

## Consequences

**积极**:
- 提交通路从"自觉"升级为"机制"，AI 无法在 spec-vc commit 之外完成 git commit
- ADR-005 的双 subagent 验证协议获得真实约束力，不再依赖 AI 行为自律
- token 过期与一次性消费保证了"每次 commit 都对应一次显式的验证流程触发"
- token 残留风险被过期时间收敛——最坏情况是 5 分钟内的旧 token 被复用

**消极**:
- 单点依赖：commit-msg hook 调用 `~/.claude/skills/spec-vc/.venv/bin/spec-vc`，spec-vc binary 损坏或 skill 路径变更会锁死整个仓库的 commit 通道
- 缺乏紧急逃生口：hotfix、CI 自动提交、bisect 等场景下无法快速绕过门禁（这一缺口由 ADR-007 单独补齐）
- 与 ADR-002 的"hook 对 skill 零依赖"原则形成张力——本机制重新引入了 hook→spec-vc binary 的运行态耦合

**中性**:
- token 过期时间硬编码为 300 秒，未来如需调整可下沉到 `.spec-vc.toml` 配置
- `.git/spec-vc-commit-token` 是运行态文件，需求 backup 与跨机器协作场景需各自独立 spec-vc 流程

## Alternatives Considered

- **SKILL.md 禁令规则**(在启动协议中明确禁止 AI 绕过 spec-vc commit)：被排除——基于自觉，AI 行为偏差时无机制约束，与"先收紧再松开"的治理哲学不符（参考 ADR-001 决策时机论述）
- **pre-commit 阶段校验**：被排除——pre-commit 早于 commit-msg，此时尚无 commit message 可读，无法与 `[ADR-NNN]` 校验形成完整语义闭环。token 校验放在 commit-msg 与既有 ADR 校验同层，逻辑更内聚
- **永久 token**(由 spec-vc commit 写入，提交才消费，不过期)：被排除——一旦某次 spec-vc commit 流程未提交完成（用户中断、subagent 失败），残留 token 会被下一次任意 `git commit` 静默复用，破坏"每次 commit 必须经过验证"的语义

## Implementation Plans

- **Plan**: doc/arch/plans/ADR-006-plan-001.md
- **Summary**: 在 commit-msg hook 中增加 token 校验逻辑——spec-vc commit 在 `.git/spec-vc-commit-token` 写入一次性 token (300s 过期)，commit-msg hook 校验 token 存在且未过期后消费。同步更新 SKILL.md 启动协议与 plan 阶段展示要求。新增 `test_commit_msg_blocks_without_token` 测试覆盖无 token 阻塞场景。
- **Pre-Validation**: 1) 当前 commit-msg hook 只校验 [ADR-NNN] 格式，不校验提交通路，git commit 可直接执行 2) SKILL.md plan 阶段无"展示计划内容到前台"要求，AI 只输出文件路径 3) 现有测试覆盖 hooks 格式校验场景但无 token 门禁测试
- **Post-Validation**: 1) 全量测试通过 2) 无 token 直接 git commit 被阻塞："未找到提交 token。请通过 spec-vc commit 流程提交代码" 3) spec-vc commit 写入 token 后 git commit 成功 4) token 一次性消费，提交后自动删除
